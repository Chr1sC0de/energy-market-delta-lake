# Ralph Curate

Use this skill to reconcile GitHub Issues with the actual repository state
before `$ralph-triage` or `$ralph-loop` act on the queue.

## Read First

- Read `CONTEXT.md` for canonical terms, especially **Subproject**,
  **Test lane**, **Local integration**, **Delivery mode**, **Integration
  target**, and **Promotion**.
- Read `docs/agents/triage-labels.md` for label vocabulary.
- Read `docs/agents/issue-tracker.md` for the issue queue contract.
- Read `docs/agents/ralph-loop.md` before recommending changes to Ralph
  workflow behavior.
- Prefer structured GitHub issue tools when available; use `gh` only when the
  available tools cannot answer the question.

## Default Safety

Default to read-only curation. Produce a curation report first and do not edit
issues, labels, comments, branches, commits, refs, or repo files unless the user
explicitly asks to apply the proposed mutations.

When applying curation, restate the exact issue numbers and mutations before
using GitHub write tools. Until a dedicated curation audit prefix exists, every
GitHub comment written by this skill must begin with:

```markdown
> *This was generated by AI during triage.*
```

Do not close issues as completed without branch evidence. If an issue might be
satisfied but the evidence is incomplete, recommend `needs-triage` or
`ready-for-human` instead of closing it.

## Curation Workflow

1. Ground in the checkout:
   - current branch and `git status --short --branch`
   - branch parity for `main`, `dev`, `origin/main`, and `origin/dev`
   - recent commits and changed files
   - relevant `rg` searches for issue titles, issue numbers, labels, flags,
     commands, docs, and acceptance-criteria terms
2. Fetch the open issue queue:
   - all open issues, labels, bodies, blockers, and comments needed for
     evidence
   - `needs-triage`, `ready-for-agent`, `ready-for-human`, unlabeled, and
     recently added issue sequences
3. Compare issue contracts to branch evidence:
   - mark implemented/satisfied only when code, docs, tests, or commit history
     support it
   - detect stale blockers, duplicated work, stale issue bodies, missing
     context, conflicting labels, and issues that should wait for earlier
     work
   - verify ready issues still have `## What to build`,
     `## Acceptance criteria`, and `## Blocked by`
4. Classify each issue:
   - no action
   - update body or current context
   - add or remove labels
   - adjust blockers
   - close completed with evidence
   - move to `needs-triage`
   - move to `ready-for-human`
5. Output an apply-ready report:
   - group recommendations by action type
   - include evidence paths, commits, comments, or searches
   - list exact proposed GitHub mutations
   - call out intentionally unchanged issues and residual uncertainty

## Label Rules

- Every open triaged issue should have one category label: `bug` or
  `enhancement`.
- Every open triaged issue should have one state label: `needs-triage`,
  `needs-info`, `ready-for-agent`, `ready-for-human`, or `wontfix`.
- Use at most one **Delivery mode** label.
- Default to `delivery-gitflow` for runtime behavior, infrastructure, Dagster,
  S3, LocalStack, cross-**Subproject** work, broad refactors, or unclear scope.
- Use `delivery-trunk` only for clearly small docs, tests, tooling, or script
  changes with low runtime risk.
- Leave blocked follow-up issues in their intended state only when the issue
  body clearly identifies blockers and the blocked state is useful for queue
  planning. Otherwise recommend `needs-triage`.
