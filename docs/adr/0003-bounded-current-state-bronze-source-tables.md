# Use Bounded Current-State Bronze Tables for Source-Table Assets

Source-table bronze assets are the GBB and VICGAS raw table assets generated by
`factories/df_from_s3_keys`. They are distinct from the scheduled
`bronze_nemweb_public_files_*` discovery/listing assets and from `unzipper_*`
assets, which extract zip payloads and archive successful zip inputs.

## Decision

Use bounded current-state Delta tables for source-table bronze assets.

Source-table bronze no longer stores append-history rows for every observed
source payload. Each ingestion run parses selected landing objects, adds
ingestion metadata, builds `surrogate_key` from the table's configured key
columns, adds `source_content_hash`, collapses the micro-batch to the maximum
`source_file` row per `surrogate_key`, and merges those current-state rows into
the target Delta table.

The merge predicate is `source.surrogate_key = target.surrogate_key`. Matched
rows update only when `target.source_content_hash IS NULL` or
`source.source_content_hash != target.source_content_hash`; unmatched source
rows insert. Target rows with no matching source row are retained. A row's
absence from a later source file is not a delete signal.

The archive bucket remains the replay source for source-table bronze rebuilds.
Rebuilds must use the bounded `aemo-replay-bronze-archive` CLI with explicit
operator intent:

- choose exactly one target scope with `--all`, `--domain`, or `--table`
- run the default dry-run first and inspect matching files, planned batches,
  total bytes, and target Delta table URI
- pass `--replace` only when the dry-run confirms the intended archive scope

Replace mode processes archive files in bounded batches. The first non-empty
batch overwrites the target table, and later non-empty batches use the same
current-state merge behavior as normal source-table bronze ingestion.

## Considered options

- Keep append-history in source-table bronze: simple to audit inside the table,
  but it makes downstream current-state reads repeatedly deduplicate rows that
  are already recoverable from archive storage.
- Use current-state bronze with archive replay: keeps serving tables bounded and
  makes archive storage the explicit replay source.
- Rebuild manually through ad hoc asset launches: works for narrow debugging,
  but does not provide dry-run evidence, bounded batch planning, or a clear
  operator intent switch before replacing Delta tables.

## Consequences

Source silver assets can treat source-table bronze as the current source state
and continue overwriting parquet snapshots from that state. Historical replay
comes from archived source files and Delta table history, not from duplicated
append-history rows inside source-table bronze tables.

No-delete-on-absence is intentional. If a source table later publishes a file
that omits a previously seen `surrogate_key`, bronze keeps the existing row
until a separate, explicit deletion design exists.

Operators must treat `--replace` as a destructive rebuild of the selected table
contents from the selected archive scope. A dry-run result is the evidence that
the target table, archive prefix, file count, batch count, and total bytes match
the intended rebuild.
